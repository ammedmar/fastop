Mathematical and package overview
=================================

``fastop`` computes cohomology operations from finite combinatorial models of
spaces.  A typical calculation has the following mathematical shape:

.. code-block:: text

   finite simplicial model X
          |
          v
   normalized cochains C*(X; F_p)
          |
          v
   chosen basis and cocycle representatives for H*(X; F_p)
          |
          +----------> Alexander--Whitney cup products
          |
          +----------> Sq^r, P^r, and beta P^r
                                |
                                v
                   projection back to H*(X; F_p)

The package constructs the cochain complex, chooses a deterministic basis for
cohomology, evaluates products or operations on representative cocycles, and
expresses the result in that same basis.  The basis is computational rather
than geometrically canonical: a printed class such as ``h^2,0`` means the
first chosen basis class in degree two, not a distinguished named generator.

The user-facing layers
----------------------

There are three supported input models.

``SimplicialComplex``
   An abstract simplicial complex generated by facets with globally named
   vertices.  This is the preferred input for an ordinary triangulation.

``DeltaComplex``
   A semi-simplicial model described by indexed face maps.  It permits loops,
   repeated faces, and quotient identifications without introducing artificial
   global vertex labels.

``SimplicialSet``
   A normalized finite simplicial set.  It retains the degeneracy information
   needed for products, symmetric powers, and compact one-vertex models.

All three expose ``dimension``, ``cells`` and ``cohomology``.  The resulting
``PrimeFieldCohomology`` object supplies Betti numbers, basis classes, cocycle
representatives, cup products, operation matrices, and operation ranks.

The :mod:`fastop.spaces` module contains standard examples and geometric
families.  In particular, ``orientable_surface`` and
``nonorientable_surface`` return compact simplicial-set models, while
``symmetric_product_of_surface`` and ``lens_space`` build the main example
families used by the project.

How the source tree reflects the mathematics
--------------------------------------------

The public modules are intentionally few:

==========================  ===================================================
Module                      Responsibility
==========================  ===================================================
``simplicial``              Abstract simplicial complexes
``delta_complex``           Indexed semi-simplicial models
``simplicial_set``          Normalized simplicial sets and their constructions
``group_action``            Strict finite actions and quotient data
``spaces``                  Standard spaces and parameterized families
``cohomology``              Cohomology groups, classes, products, and operations
==========================  ===================================================

Modules whose names begin with an underscore are implementation details.  They
contain sparse linear algebra, universal Steenrod formulas, cochain evaluators,
catalog data, and the optional native accelerator.  Ordinary calculations
should not import them.

What is computed lazily
-----------------------

Cohomology data is built degree by degree.  Asking for ``H^d`` constructs only
the adjacent linear-algebra data needed in that degree.  Symmetric powers are
also degree-lazy: their total ``f_vector`` can be predicted cheaply, while
``cells(d)`` materializes the requested layer.

This distinction matters for large examples.  A model may have millions of
nondegenerate simplices in total while a particular operation requires only
its source and target degrees.  Use ``symmetric_power_f_vector`` before
constructing a large symmetric power.

Public and advanced controls
----------------------------

The normal interface chooses operation formulas and evaluation strategies
automatically.  Formula-source and evaluator-selection controls exist for
validation and benchmarking, but they are implementation-facing diagnostics
rather than part of the basic mathematical workflow.

Continue with :doc:`input-models` to choose a representation, then
:doc:`operations` for cohomology products and Steenrod operations.
